# 仓库源码解析落地 Markdown

## 目标

给定一个 Git 仓库（优先 GitHub URL），在本地拉取/读取源码，梳理目录结构与关键调用链，生成一份中文 `.md` 源码解析文档并写入当前工作区。

文档要求：
- 解析内容以“读懂源码”为中心（模块分层、关键数据结构、关键流程、主要边界条件）。
- 只要出现“引用源码”的场景：要么不写引用，要么直接把对应源码片段贴进 `.md`（fenced code block）。
- 不在正文中放可点击的本地 `file:///...` 链接；如需定位，用纯文本位置标注：`src/xxx.ts:L10-L42`。
- 文档开头必须包含真实仓库地址（例如 `https://github.com/chenglou/pretext`），不要写本地克隆路径。

## 何时调用

当用户提出以下诉求时调用：
- “读这个仓库源码/帮我分析这个仓库”
- “生成一份源码解析/代码导览/架构解读”
- “把解析结果输出成 .md 文件”
- “可以启动多个 subagent 去调研”

## 入参（建议）

- `repo_url`（必填）：仓库地址（如 GitHub URL）。
- `output_path`（可选）：输出 `.md` 的工作区路径。
  - 默认：`<repo-name>-source-analysis.md`
- `focus`（可选）：关注点列表（如 `["核心 API","换行算法","缓存策略","构建与 demo"]`）。
  - 默认：优先覆盖 `API/核心模块/构建验证` 三块。
- `depth`（可选）：`"overview" | "deep"`，控制摘录代码数量与解释深度。
  - 默认：`"overview"`

## 工作流

### 1）拉取仓库并勘察目录

- 将仓库 clone 到工作区的 `_ext/<repo-name>`（或同等隔离目录）。
- 列出顶层目录结构，识别：
  - 入口文件/导出文件（如 `src/index.ts`、`src/layout.ts`、`package.json` 的 `main/exports`）
  - 核心模块目录（如 `src/`、`lib/`、`packages/`）
  - demo/脚本/测试（如 `pages/`、`scripts/`、`tests/`）
- 阅读 `README.md`/`DEVELOPMENT.md`/`package.json`，形成“仓库是做什么的”与“怎么跑起来”的骨架信息。

### 2）并行调研（必须使用多个 subagent）

根据 repo 结构，把调研拆成互不冲突的子任务，并行执行：
- 子任务 A：公共 API 与入口文件（exports、main、types、对外函数）
- 子任务 B：核心算法/关键模块（如 layout、parser、engine、scheduler 等）
- 子任务 C：性能与缓存策略（memoization、LRU、WeakMap、benchmark）
- 子任务 D：构建/测试/脚本与 demo（如何验证正确性、如何发布）

每个子任务输出必须包含：
- 关键文件清单（文件路径列表）
- 关键函数/类型名
- 关键逻辑的“原理解释”
- 至少 1 段源码摘录（贴到最终 md）

### 3）精读与交叉校验

- 对照子任务结论，补齐调用链的缺口（例如：API -> prepare -> analyze -> measure -> layout）。
- 对关键结论做“源码对照验证”：若文中提到某个规则/分支，必须能在源码中找到对应实现，并摘录进文档。

### 4）生成 Markdown 文档并写入

输出结构建议：

```md
# <RepoName> 源码解析

## 摘要
<从 README.md 提炼：仓库做什么、解决什么问题、核心卖点与用法>

仓库地址：<repo_url>

## 目录与模块分层
<模块职责说明>

## 核心 API 与调用链
<入口函数、主要数据结构、流转过程>

## 关键实现细节
<算法、边界条件、浏览器/平台差异等>

## 构建与验证
<如何跑 demo、如何跑测试、如何生成发布产物>
```

写作规范：
- 所有源码摘录用 fenced code block，语言标签使用仓库主要语言（如 `ts`/`js`/`go`）。
- 摘录要短（优先 20–120 行），且与解释一一对应。
- 避免复制整文件；只截取能支撑结论的关键片段。

### 5）安全与合规

- 不要在文档中输出 token、cookie、个人信息、私有 URL、密钥等。
- 若仓库包含凭证示例（如 `.env`），只描述用途，不贴原文。

## 示例触发语句

- “帮我读这个仓库的源码并生成一份源码解析.md：<repo_url>”
- “我经常让你读源码并生成文档，以后用这个流程来做”
